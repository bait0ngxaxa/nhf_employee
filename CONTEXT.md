# NHF Employee

บริบทนี้นิยามภาษาธุรกิจของระบบงานภายใน NHF สำหรับข้อมูลพนักงาน ระบบลา งานบริการ IT สต็อก การแจ้งเตือน และรายงานที่เกี่ยวข้อง

## Language

**บันทึกการใช้งาน**:
ประวัติเหตุการณ์สำคัญในระบบที่ต้องอ่านได้ว่าใครทำอะไรกับข้อมูลอะไรและมีรายละเอียดสำคัญอะไร โดยข้อความหลักต้องเป็นภาษาไทยที่ผู้ดูแลระบบหรือผู้เกี่ยวข้องอ่านเข้าใจได้โดยไม่ต้องตีความค่าทางเทคนิค
_Avoid_: audit log ดิบ, action code, ประเภทข้อมูลอย่างเดียว

**รายละเอียดบันทึกการใช้งาน**:
ข้อความสรุปข้อมูลสำคัญของเหตุการณ์ในบันทึกการใช้งาน โดยแสดงเฉพาะสิ่งที่ช่วยตรวจสอบหรือเข้าใจผลกระทบ เช่น รายการที่เกี่ยวข้อง ค่าเดิมและค่าใหม่ เหตุผล ช่วงวันที่ จำนวนวัน หรือจำนวนสต็อก ไม่ใช่การแสดงข้อมูลดิบทั้งหมด
_Avoid_: JSON ดิบ, รายละเอียดทางเทคนิคทั้งหมด, user agent

**บันทึกการใช้งานย้อนหลัง**:
บันทึกการใช้งานที่ถูกสร้างก่อนการปรับปรุงรูปแบบการแสดงผล ต้องแสดงรายละเอียดภาษาไทยให้ดีที่สุดจากข้อมูลที่มีอยู่ และใช้ข้อความสรุปทั่วไปเมื่อข้อมูลเดิมไม่เพียงพอ
_Avoid_: migrate ทุกแถว, แสดง raw data เพราะเป็นข้อมูลเก่า

**ผู้อ่านบันทึกการใช้งาน**:
แอดมิน HR หรือเจ้าหน้าที่ที่ดูแลระบบซึ่งต้องตรวจสอบเหตุการณ์ในระบบด้วยภาษาธุรกิจ ไม่ใช่นักพัฒนาที่อ่านรหัสเหตุการณ์หรือโครงสร้างข้อมูลดิบ
_Avoid_: developer เป็นผู้อ่านหลัก, technical log viewer

**เหตุการณ์ในบันทึกการใช้งาน**:
ข้อความหลักของแต่ละแถวในบันทึกการใช้งานที่รวมประเภทเหตุการณ์กับรายละเอียดภาษาไทยไว้ด้วยกัน เพื่อให้ผู้อ่านเข้าใจเหตุการณ์ได้จากแถวเดียวโดยไม่ต้องเปิดข้อมูลดิบ
_Avoid_: แยก action code กับ JSON details, อ่านจากหลายคอลัมน์ถึงจะเข้าใจ

**ข้อมูลอ่อนไหวในบันทึกการใช้งาน**:
ข้อมูลที่ไม่ควรแสดงในรายละเอียดบันทึกการใช้งานแม้ถูกบันทึกไว้เพื่อการตรวจสอบ เช่น รหัสผ่าน token session id cookie หรือข้อมูลเทคนิคที่ไม่จำเป็นต่อการตรวจสอบเหตุการณ์ ส่วนชื่อ อีเมล วันที่ เหตุผลลา และจำนวนสต็อกแสดงได้เมื่อเกี่ยวข้องกับงานของผู้อ่านบันทึกการใช้งาน
_Avoid_: แสดง secret ดิบ, ซ่อนข้อมูลธุรกิจที่จำเป็นทั้งหมด

**คำขอลา**:
รายการที่พนักงานยื่นเพื่อขอใช้สิทธิ์ลาในช่วงวันที่และประเภทการลาที่ระบุ โดยต้องผ่านการพิจารณาจากผู้อนุมัติก่อนถือว่าใช้สิทธิ์จริง
_Avoid_: ใบลา, leave record

**เหตุการณ์แจ้งเตือนการลา**:
เหตุการณ์สำคัญในวงจรคำขอลาที่ต้องแจ้งผู้เกี่ยวข้อง เช่น ส่งคำขอใหม่, อนุมัติ, ไม่อนุมัติ, ยกเลิกคำขอที่ยังรออนุมัติ, แจ้งไม่ได้ใช้วันลา, และยืนยันไม่ได้ใช้วันลา ส่วนลาย้อนหลังกรณีฉุกเฉินและคำขอลาเกินโควต้ากรณีพิเศษเป็นข้อมูลประกอบในเหตุการณ์ส่งคำขอใหม่ ไม่ใช่เหตุการณ์แจ้งเตือนแยก
_Avoid_: ประเภทการลา, leave type

**ผู้รับแจ้งเตือนการลา**:
ผู้เกี่ยวข้องที่ต้องได้รับข้อมูลจากเหตุการณ์แจ้งเตือนการลา โดยปกติคือผู้ที่ต้องดำเนินการต่อหรือผู้ที่ได้รับผลลัพธ์จากการพิจารณา ส่วนผู้กระทำเหตุการณ์เองได้รับเพียงการยืนยันในระบบเมื่อจำเป็นเพื่อลดการแจ้งเตือนซ้ำ
_Avoid_: ผู้เกี่ยวข้องทั้งหมด, everyone

**ผู้อนุมัติที่ตั้งค่าแล้ว**:
พนักงานที่ถูกกำหนดเป็นผู้อนุมัติของพนักงานอีกคนและมีบัญชีผู้ใช้ที่ใช้งานอยู่ในระบบ จึงสามารถรับการแจ้งเตือนในระบบและดำเนินการอนุมัติได้
_Avoid_: มีหัวหน้าในข้อมูลพนักงานเท่านั้น, managerId only

**การแจ้งเตือนซ้ำ**:
การแจ้งเตือนในระบบที่ถูกสร้างมากกว่าหนึ่งครั้งให้ผู้รับคนเดิมจากเหตุการณ์การลาเดียวกัน ต้องป้องกันด้วยตัวตนของผู้รับ ประเภทการแจ้งเตือน และคำขอลาที่อ้างอิง
_Avoid_: retry, ส่งซ้ำทั้งหมด

**ลาย้อนหลังกรณีฉุกเฉิน**:
คำขอลาที่พนักงานสร้างหลังวันที่ลาจริง เพราะมีเหตุฉุกเฉินจนไม่สามารถยื่นลาล่วงหน้าหรือยื่นในวันนั้นได้ ใช้ได้กับการลาทุกประเภท ระบบระบุคำขอย้อนหลังจากวันที่เริ่มลาที่น้อยกว่าวันปัจจุบันโดยอัตโนมัติ คำขอนี้ยังต้องผ่านผู้อนุมัติตาม flow ปกติ ย้อนหลังได้ไม่เกิน 7 วันปฏิทินนับจากวันที่เริ่มลา และไม่บังคับแนบหลักฐาน
_Avoid_: แก้ย้อนหลัง, HR เพิ่มวันลาให้

**เหตุผลฉุกเฉิน**:
ข้อความอธิบายว่าทำไมพนักงานจึงไม่สามารถยื่นคำขอลาได้ตามเวลาปกติ ใช้ประกอบการพิจารณาคำขอลาย้อนหลังกรณีฉุกเฉิน
_Avoid_: หมายเหตุ, comment

**วันลาสุทธิ**:
จำนวนวันลาที่นำไปตัดโควตา โดยนับเฉพาะวันทำงานจันทร์ถึงศุกร์และรองรับครึ่งวันได้ คำขอลาย้อนหลังกรณีฉุกเฉินใช้กฎการนับวันลาสุทธิเดียวกับคำขอลาปกติ
_Avoid_: จำนวนวันตามปฏิทิน, calendar days

**การปรับรายการลาหลังอนุมัติ**:
การบันทึกเหตุผลเมื่อคำขอลาได้รับอนุมัติแล้ว แต่วันลาจริงพนักงานไม่ได้ลา พนักงานเป็นผู้แจ้งพร้อมโน๊ตหลังวันสิ้นสุดการลาผ่านไปแล้ว และผู้อนุมัติต้องยืนยันก่อนระบบคืนโควต้าวันลาตามวันลาสุทธิของคำขอเดิม
_Avoid_: แค่โน๊ต, comment หลังอนุมัติ

**แจ้งไม่ได้ใช้วันลา**:
เหตุการณ์ที่พนักงานแจ้งหลังคำขอลาได้รับอนุมัติและวันสิ้นสุดการลาผ่านไปแล้วว่าไม่ได้ลาในวันจริง เหตุการณ์นี้รอผู้อนุมัติยืนยันก่อนจึงถือว่าสิ้นสุด
_Avoid_: ขอคืนโควต้า, quota refund request

**ไม่ได้ใช้วันลา**:
สถานะของคำขอลาที่เคยอนุมัติแล้ว แต่ภายหลังผู้อนุมัติยืนยันว่าพนักงานไม่ได้ลาในวันจริง สถานะนี้แยกจากการยกเลิกคำขอที่ยังรออนุมัติ ต้องยังแสดงในรายงานการลาแต่ไม่ถูกนับเป็นวันลาที่ใช้สุทธิ
_Avoid_: ยกเลิก, cancelled

**คำขอลาเกินโควต้ากรณีพิเศษ**:
คำขอลาที่พนักงานยื่นเมื่อใช้โควต้าประเภทนั้นเต็มแล้วหรือจำนวนวันที่ขอเกินโควต้าคงเหลือ ใช้ได้กับการลาทุกประเภทและเกิดร่วมกับลาย้อนหลังกรณีฉุกเฉินได้ ผู้อนุมัติเดิมของพนักงานสามารถอนุมัติได้โดยไม่ต้องมี HR approval รอบสอง เมื่ออนุมัติแล้วไม่เพิ่มโควต้าเดิมและไม่ยืมโควต้าปีหน้า แต่ให้ยอดใช้ปีปัจจุบันเกินโควต้าได้
_Avoid_: เพิ่มโควต้า, leave quota top-up

**เหตุผลพิเศษ**:
ข้อความอธิบายว่าทำไมคำขอลาที่เกินโควต้าควรได้รับการอนุมัติ ใช้ประกอบการพิจารณาคำขอลาเกินโควต้ากรณีพิเศษและแยกจากเหตุผลการลา
_Avoid_: เหตุผลการลา, หมายเหตุทั่วไป
